Доклад Patrice Roy: многие способы убить орка

Доклад раскрывает архитектурную эволюцию игровых сущностей в C++: от простого наследования до modern C++ без runtime-polymorphism.

Ссылка на видеодоклад GitHub Pages: https://noniari.github.io

О спикере

Patrice Roy — инженер C++ и докладчик CppCon, который использует игровой пример с героями и орками, чтобы рассказать не о играх, а об архитектуре кода.

В начале доклада он прямо говорит, что не является игровым программистом, а герои/орки служат только для иллюстрации архитектурных решений.

  • Доклад на CppCon 2024 посвящён дизайну кода, а не игровым механикам.
  • Показывает, как private inheritance, interfaces, templates и concepts влияют на поддерживаемость.
  • Основная мысль: выбор архитектуры должен зависеть от задачи, а не от привычного шаблона.

Введение

В начале доклада Patrice сразу уточняет: это не доклад для игровых программистов, а речь о проектировании кода. Он использует метафору орков и героев, чтобы показать разные архитектурные подходы.

  • видео начинается с объяснения, что код в докладе компилирует и всё доступно в слайдах;
  • герои и орки — пример для обсуждения ответственности классов и разделения обязанностей;
  • внимание смещено на то, как разные способы реализации влияют на расширяемость и сложность.

1. Базовая игровая модель

Герой и орк наследуются от Character, но задача показывает, что простая система быстро становится неудобной.

Автор использует private inheritance, чтобы скрыть реализацию Character и избежать slicing — но это усложняет polymorphism.

Добавление метода attack в базовый класс ведёт к feature creep. Это служит примером, как не стоит расширять ответственность общей сущности.

Примеры кода

В докладе объясняется, что наследование и виртуальные функции влияют не только на синтаксис программы, но и на то, как объект устроен в памяти. Ниже приведён пример обычного наследования, который помогает понять тему.

Обычное наследование

Здесь класс Dog наследуется от класса Animal. Объект Dog содержит часть Animal и добавляет своё поле.

class Animal {
public:
    int age;

    void eat() {
        // общее действие для всех животных
    }
};

class Dog : public Animal {
public:
    int weight;

    void bark() {
        // действие, которое есть только у собаки
    }
};

int main() {
    Dog dog;

    dog.age = 3;
    dog.weight = 15;

    dog.eat();
    dog.bark();
}

В этом примере нет виртуальных функций, поэтому вызовы методов достаточно простые. Компилятор заранее понимает, какую функцию нужно вызвать.

2. Damageable и Bellicose

Разделение обязанностей делает структуру легче расширяемой и уменьшает связность.

Damageable

Интерфейс для любого объекта, который может получить урон.

struct Damageable {
    virtual void suffer(int damage) = 0;
};

Bellicose

Интерфейс атаки разделён на отдельный тип, чтобы класс Character не обременялся лишней логикой.

struct Bellicose {
    virtual void hit(Damageable& target) = 0;
};

Armor

Броня оформлена как отдельный объект композиции, а не как часть базовой сущности.

class Armor {
public:
    int protection() const;
};

3. Templates и concepts

Переход к modern C++ показывает, что поведение важнее иерархии.

Concepts

Поведенческие требования задаются на уровне типов без virtual-функций.

std::variant

Разные игровые сущности хранятся в едином контейнере без runtime-полиморфизма.

Performance

Variant-подход уменьшает размер объекта и ускоряет вызов методов.

4. Выводы

Доклад подчеркивает, что архитектуру нужно выбирать под задачу, а не под привычный шаблон.

  • Не добавляйте функциональность в базовый класс ради быстрого исправления.
  • Современному C++ доступны inheritance, templates, concepts и variant.
  • Каждый подход имеет свои преимущества и цену поддержки.
