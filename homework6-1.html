Доклад Patrice Roy: многие способы убить орка

Доклад раскрывает архитектурную эволюцию игровых сущностей в C++: от простого наследования до modern C++ без runtime-polymorphism.

О спикере

Patrice Roy — инженер C++ и специалист по игровой архитектуре, который рассказывает о проектных решениях, важности разделения ответственности и современном использовании языка для игровых систем.

Patrice выступал на CppCon 2024 с докладом на тему того, как дизайн кода влияет на расширяемость и производительность игровых механик.

  • Опыт с C++ более 10 лет.
  • Специализируется на game development и архитектуре систем.
  • Использует композицию, templates и std::variant в современных проектах.

Введение

Доклад объясняет, почему архитектура важнее выбора конкретной игровой механики. На примере героя и орка обсуждаются:

  • когда наследование делает систему жёсткой;
  • почему классы не должны брать на себя функции, которые им не соответствуют;
  • как modern C++ позволяет выбирать поведение по возможностям, а не по иерархии.

1. Базовая игровая модель

Герой и орк наследуются от Character, но задача показывает, что простая система быстро становится неудобной.

Автор использует private inheritance, чтобы скрыть реализацию Character и избежать slicing — но это усложняет polymorphism.

Добавление метода attack в базовый класс ведёт к feature creep. Это служит примером, как не стоит расширять ответственность общей сущности.

2. Damageable и Bellicose

Разделение обязанностей делает структуру легче расширяемой и уменьшает связность.

Damageable

Интерфейс для любого объекта, который может получить урон.

struct Damageable {
    virtual void suffer(int damage) = 0;
};

Bellicose

Интерфейс атаки разделён на отдельный тип, чтобы класс Character не обременялся лишней логикой.

struct Bellicose {
    virtual void hit(Damageable& target) = 0;
};

Armor

Броня оформлена как отдельный объект композиции, а не как часть базовой сущности.

class Armor {
public:
    int protection() const;
};

3. Templates и concepts

Переход к modern C++ показывает, что поведение важнее иерархии.

Concepts

Поведенческие требования задаются на уровне типов без virtual-функций.

std::variant

Разные игровые сущности хранятся в едином контейнере без runtime-полиморфизма.

Performance

Variant-подход уменьшает размер объекта и ускоряет вызов методов.

4. Выводы

Доклад подчеркивает, что архитектуру нужно выбирать под задачу, а не под привычный шаблон.

  • Не добавляйте функциональность в базовый класс ради быстрого исправления.
  • Современному C++ доступны inheritance, templates, concepts и variant.
  • Каждый подход имеет свои преимущества и цену поддержки.
