Our Profiling and Beyond

Оптимизация производительности в игровых движках

Доклад раскрывает архитектуру собственного инструментального профайлера в крупных игровых проектах. Авторы показывают не только теорию профилирования CPU, GPU, памяти и файловой системы, но и реальные инженерные решения: hook-системы, сбор GPU-таймингов, анализ VRAM и ETW-трейсинг Windows.

Введение

Почему профилирование критически важно для игр:

Реальное время

Игры — это системы реального времени. Каждый кадр должен укладываться в ограниченное время: 60 FPS → ~16.6 мс на кадр; 120 FPS → ~8.3 мс.

Видимость проблем

Если какая-либо система начинает работать медленнее — игрок сразу замечает лаги, фризы или просадки FPS.

Комплексность

Проблемы производительности могут создавать не только программисты, но и художники: большие текстуры, тяжёлые модели, лишние draw calls.

Ключевое сообщение доклада

Производительность вездесуща и является краеугольным камнем при разработке видеоигр. Инструментами профилирования должны пользоваться все, начиная от программистов движка, вылавливающих «спайки» и оптимизирующих код под самые разнообразные конфигурации, которые встречаются у игроков, заканчивая художниками и геймдизайнерами, следящими за техническим состоянием игрового контента.

О чём рассказано в докладе

Спикеры рассказали про особенности профилирования приложений реального времени, о том, как проверка производительности стала частью процесса разработки, и самое интересное — про внутреннее устройство собственного профайлера. Детально коснулись того, как измеряют производительность CPU и GPU, RAM/VRAM, работу с диском и счетчиками. Поделились трюками, которые используют, чтобы собрать нужную информацию из операционной системы и видеодрайвера (даже если они не хотят ей делиться), а также подходами для локализации проблем в сложном и непредсказуемом окружении.

1. Виды профилирования

Авторы показывают, что обычных профайлеров недостаточно. Существуют два основных подхода:

Sampling Profiling

Sampling-профилирование работает так: приложение периодически останавливают, снимают stack trace, смотрят, где чаще всего находится CPU.

✓ Плюсы

  • Легко использовать
  • Минимальная интеграция

✗ Минусы

  • Результаты не всегда детерминированы
  • Плохо подходит для многопоточности
  • Неудобно сравнивать между билдами

Instrumentation Profiling

В инструментальном профилировании разработчики сами расставляют метки в коде.

PROFILE_SCOPE("Physics");

Профайлер точно знает: когда началась зона, когда закончилась, какой поток её выполнял.

✓ Плюсы

  • Высокая точность
  • Удобно сравнивать билды
  • Хорошо работает в многопоточности
  • Подходит даже для скриптовых языков

✗ Минусы

  • Требуется интеграция в движок

Вывод: Для больших игровых проектов нужен собственный инструментальный профайлер, который почти не влияет на производительность, умеет работать с многопоточностью и собирает CPU и GPU тайминги.

2. Архитектура собственного профайлера

Авторы реализовали многоуровневую систему профилирования. Основные слои:

Слои архитектуры

В коде движка расставляются маркеры:

PROFILE_SCOPE("Render");
PROFILE_SCOPE("Physics");
PROFILE_SCOPE("AI");

Макросы позволяют:

  • Включать/выключать профилирование
  • Менять backend
  • Подключать разные профайлеры

Каждый logger отвечает за свой тип данных:

  • CPU — измерение времени выполнения кода
  • GPU — timestamp queries, pipeline analysis
  • RAM — отслеживание аллокаций
  • VRAM — анализ памяти видеокарты
  • IO — мониторинг файловых операций

После записи данные обрабатываются:

  • Сохраняются в файлы
  • Отправляются по сети
  • Рисуются на графиках
  • Агрегируются для regression testing

3. CPU Profiling

Как измерить производительность процессора с минимальным оверхедом

Time Stamp Counter (RDTSC)

Для измерения времени используется инструкция процессора RDTSC, которая возвращает количество тактов CPU.

✓ Плюсы

  • Очень высокая точность
  • Минимальный overhead

✗ Минусы

  • Частоту приходится калибровать вручную
  • Может быть нестабильна на некоторых процессорах

RAII Профилирование

Вместо ручного start/end используется RAII (Resource Acquisition Is Initialization) — современный C++ подход для автоматического управления ресурсами.

class ProfileScope {
public:
    ProfileScope(const char* name) {
        StartTime = ReadTSC();
    }
    
    ~ProfileScope() {
        uint64_t End = ReadTSC();
        SaveResult(End - StartTime);
    }
    
private:
    uint64_t StartTime;
};

Использование:

void UpdatePhysics() {
    ProfileScope scope("Physics");
    Simulate();
}

Такой подход автоматически измеряет время выполнения блока при разрушении объекта.

Многопоточность

Профайлер отдельно отслеживает:

  • Worker threads (рабочие потоки)
  • Main thread (основной поток)
  • Background jobs (фоновые задачи)

Для этого каждому потоку задаётся имя. Windows не предоставляет удобный API для naming threads, поэтому используется специальный exception-based hack. Это позволяет видеть имена потоков в профайлере и в Visual Studio debugger.

4. GPU Profiling и ETW

GPU profiling намного сложнее CPU profiling, так как GPU работает асинхронно и параллельно с CPU.

GPU Timestamp Queries

Для GPU используются timestamp queries. Схема работы:

  1. GPU получает команду
  2. В pipeline вставляется query
  3. GPU записывает timestamp
  4. Через несколько кадров результат считывается с GPU памяти

GPU Cyclic Buffer

Чтобы не тормозить GPU, используется ring buffer. Пока GPU пишет новые тайминги, CPU читает старые. Это минимизирует stalls и позволяет профайлеру работать без заметного влияния на производительность.

Hooking DirectX

Авторы внедряются в DirectX через vtable hooking. Это позволяет перехватывать вызовы DirectX и собирать:

  • Draw calls
  • Dispatch calls
  • Barriers
  • Resource creation
  • State changes

Профайлер показывает:

  • Количество примитивов
  • Пиксели
  • Vertex shader load
  • Pixel shader load
  • Render targets
  • Pipeline bottlenecks

ETW — Event Tracing for Windows

Авторы используют ETW — инструмент ядра Windows для отслеживания системных событий на уровне ядра.

Что видно в профайлере через ETW:

  • Context switches — смена активных потоков
  • Scheduler — решения планировщика ОС
  • Disk activity — работа дисковой подсистемы
  • Работа драйверов — взаимодействие с видеодрайвером

Практический пример:

Можно увидеть, что Chrome занял CPU, главный поток игры был остановлен, и FPS просел. ETW позволяет локализовать такие проблемы на уровне ОС.

5. Memory Profiling: RAM и VRAM

Профилирование памяти нужно для поиска утечек, allocation spikes и причин фризов.

RAM Profiling

В движке переопределяются глобальные функции аллокации:

void* Allocate(SizeT Size) {
    Profiling::TrackAllocation(
        GGenericHeap,
        Ptr,
        Size);
    return Ptr;
}

Отслеживание аллокаций

Каждая аллокация записывается в профайлер вместе с:

  • Stack trace — где в коде произошла аллокация
  • Heap — на каком heap'е
  • Размер аллокации

Это помогает искать утечки памяти, временные allocations и fragmentation.

VRAM Profiling

GPU-память анализируется отдельно через DirectX API:

struct GPUMemoryStats {
    struct Stats {
        SizeT Usage;      // Текущее использование
        SizeT Budget;     // Максимальный лимит
        Atomic64 Resource;
    };
    Stats CpuMemory;
    Stats GpuMemory;
};

GPU Resources Tree

Профайлер строит дерево ресурсов:

GPU Resources
├── Textures
├── Models
├── Buffers
└── Render Targets

Это помогает художникам находить слишком тяжёлые ресурсы и оптимизировать контент.

Allocation Hooking via Import Table

Авторы используют import table hooking — внедрение в таблицу импорта Windows API функций.

LPVOID WINAPI OurVirtualAlloc(...) {
    Profiling::TrackAllocation(
        GVirtualMemoryHeap,
        lpAddress,
        dwSize);
    
    return GVirtualMemoryOriginalFunctions
        .VirtualAlloc(...);
}

Профайлер может видеть даже системные аллокации Windows, что позволяет найти проблемы на более глубоком уровне.

6. File IO Profiling

Игры постоянно читают данные: текстуры, модели, карты, аудио. Профилирование диска помогает найти bottleneck'и.

Отслеживаемые операции

  • Create
  • Read
  • Write
  • Rename
  • Delete
  • Flush
  • FSCTL

Найденные проблемы

  • Слишком маленькие чтения
  • Плохой streaming
  • Bottleneck диска
  • Проблемы декомпрессии

ETW для диска

ETW позволяет увидеть:

  • Кто использует диск
  • Какие процессы мешают игре
  • Сколько времени занимает IO

7. Data Plots и Performance QA

Профайлер собирает счётчики производительности и строит графики.

Примеры счётчиков

PROFILE_COUNTER(
    "Triangles.Terrain",
    GStatsTriByType[EStats::TERRAIN]);

PROFILE_COUNTER(
    "Particles.PixelsDrawn",
    PixelsDrawn);

Отображаемые метрики

  • Triangles
  • Particles
  • Visible lights
  • RAM usage
  • VRAM usage
  • FPS
  • IO stats

Агрегация данных

  • Average
  • Min
  • Max
  • Median
  • Sum

Помогает находить регрессии между сборками.

Performance QA

Performance QA использует профайлер для проверки:

  • Не ухудшился ли FPS
  • Не выросла ли память
  • Не появились ли spikes

Основные выводы доклада

Профайлер — часть движка

Обычных external tools недостаточно. Для больших проектов нужна собственная система профилирования, глубоко интегрированная в движок.

Instrumentation > Sampling

Instrumentation profiling лучше подходит для игр — он детерминирован, точен и удобен для regression testing.

GPU так же важен, как CPU

Даже если CPU быстрый — bottleneck может быть в GPU pipeline. Нужно профилировать обе стороны.

Memory profiling обязателен

Проблемы памяти часто вызывают spikes, stutters, page faults и фризы. Отслеживание аллокаций критически важно.

ETW — мощный инструмент

ETW позволяет видеть scheduler, disk IO, context switches и системные события. Это помогает локализовать проблемы на уровне ОС.

Профайлер для всех

Инструментами профилирования должны пользоваться программисты, художники и QA — каждый может найти проблемы в своей области.

Главная идея: Профайлер должен быть точным, быстрым, минимально intrusive и удобным для программистов, QA и художников. Авторы показывают, как современные игровые движки используют low-level C++, Windows internals, DirectX hooking, ETW, instrumentation profiling и memory tracking для достижения максимальной производительности.
